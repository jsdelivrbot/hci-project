High-Fidelity Prototype

User Manual

The mobile game is developed with both Android and iOS in mind. However, due to restrictions on iOS development, a test release of the application is only feasible on the Android platform. The test will be conducted on a real smartphone. This is to provide an experience as close as possible to the target experience. It allows the user to feel the prototype on a physical phone, touch the screen, evaluate the layout, and place themselves in the context of the game. The steps below detail how to install the prototype of Pass the Phone on your own Android phone. Alternatively, a phone from the development team already running the application can be provided for testing.

Installing Pass the Phone on an Android phone

  1. Open the Settings application, navigate to Security, and turn on the “Unknown Sources” option. This will allow you to install an application that is not from the Play Store onto your phone.
  2. Download the prototype (.apk).
  3. Open the downloaded .apk file. Your device should prompt you to allow the installation.
  4. Pass the Phone is now installed on your phone! You can open it just like you would any other application, and start using it.

Again, if this is not a possibility for the testing team, a phone that has gone through the above steps can be provided.

Using Pass the Phone

The game aims to be as straightforward as possible. We want the learning curve to be low, as is characteristic of mobile games. Before being able to play, a brief setup period is required. At this stage, the user is asked to enter the following information, in sequence: a group name, player names, a game mode, the members of each teams (if team vs. team mode chosen), and a subject. Afterwards, the game can be started. The person holding the phone will be prompted to ask a question to another player, and pass the phone to that player so they can answer. The game unfolds from there.

Design Evolution

After testing the low-fidelity prototype, several possible improvements to our design arose. The conversion from a paper medium to the actual smartphone also induced some change in the design. Here are some of the more important changes that were made. Screenshots of the prototype illustrating these changes are included in the following section.

1. Addition of a back button

An important addition to the design that was not included at first is a back button, located in the top left of the application during the setup. During testing, we noticed that the desire to move back in the setup, for example to change game mode or a team member’s name, manifested itself more than we expected. The back button is unobtrusive, placed in the header and to the left just as described in the design guidelines of both Android and iOS. Since it is right where users expect, this small addition instantly communicates its purpose and makes the setup portion of the game easier to use.

2. Addition of a setup progress indicator

The progress indicator serves to show at which stage of the setup the user is in a general way. It is displayed as a line with a highlighted part, which creates a progress metaphor. This type of display element is often used in mobile applications, and should feel right at home in our prototype. It ties in with the usability goals we had fixed for the setup: it makes it easier to use, and is a good hint of whether the setup is taking too long.

3. Group name clarification

During the testing of our low-fidelity prototype, users reported that the text box for entering a "group name" was unclear, since they didn't know what the purpose of this name was. We added a small explanation above the group name text field in an attempt to make this clearer, and included the player names on this page to create a conceptual link between a group and its members.

4. Redesign of the layout of the Summary of Results page

After a set of rounds, we want users in a same group to be able to play a new game quickly, without entering the long setup process again. This requires several options: starting a new game right away, or start a new game with a different subject, or with different players. These options were included on the same page as the results after a game (each player's points).

User Population

The target user base for this game is groups of friends that know each other, and are in a situation where everyone is on their phone and little social interaction is ongoing among the group. As such, there is no required expertise or age to test the game. However, the user base should already be familiar with smartphones. The optimal testing environment would be to give the application to a group of people. This will allow the game to go through normally, with the phone being passed around. However, the application can also be tested on a single person, to see whether they understand the conceptual flow of the game and whether the given directions are clear.

Usability Goals

For the game to be successful, we set 6 usability goals, separated between the initial setup required before the game can be played, and then the game itself. They were refined from the low-fidelity usability goals, and an additional game goal was added.

Initial Setup Goals

1. The setup is easy to use

We want the setup to be painless, so it should employ existing conventions used by mobile application to allow the user to apply known concepts to our interface. The hesitation of users, the mistakes they make, and feedback they give will be used to determine whether we’re on the right track. A new user should not make more than 2 errors throughout the setup. Errors include entering the wrong information in a field, or selecting a wrong option relating to the task at hand.

2. The setup is not too long

This goes hand in hand with the previous goal. Before playing, some information is necessary for the game to progress properly: a group name, IDs for each player, and the selection of a subject and of a game mode. Since we want the game to get people off their phones, we don’t want the person doing the initial setup on the phone that will be passed around to be isolated for a significant period of time. The setup should take no more than 3 minutes.

3. The setup does not repeat too often

Since the setup is a significant barrier between opening the game and starting to play, we aim at minimizing its occurrence. For example, if users want to start a new game with a new subject and the same group right after finishing a game, they should not have to go through the entire setup again. The tests will have to note what actions users want to do that would lead them to the setup screen, and whether the current means of avoiding it are successful. The setup screen should not be entered in its entirety more than once for a given group, even if a new round is started. This goal is also not met if users do not make use of the options provided to start a new game and instead instinctively reset the application to the beginning to do so.

Game Goals

4. The game flows well from person to person

Since the game involves a single phone being passed around, the steps to be taken by the players to move the game forward need to be explicit. The aim is to avoid confusion on how to proceed (for example, who to pass the phone to, or who should ask a question to whom). If new users ask more than 3 questions related to the progression of the game, then this goal is not met.

5. All players participate in the the game

We want the game to be playable with a single phone, but also to engage all members of the game. We have thought of several means to do this: the act of passing the phone around, a timer, text-to-speech for elements that only one person sees, and others. We will have to see whether these are effective at keeping people playing as a group, without members being left hanging for long periods of time. If a player does not receive the phone as often as the other players, or if they do not ask questions as often, then this goal is not met.

6. The length of the game is adequate

If the game is too short, the group might spend too much time in the setup part of the application, choosing subjects, and restarting. If the game is too long, then players may lose interest. This is subjective, but will most likely surface in user feedback.

7. The game is a fun solution when people notice that they are all on their smartphone

The game relies on the users to access it from the application icon when they notice that their group is not socializing. To ensure that users actually use it, it should be fun, rather than requiring too much effort to get going. The positivity of feedback received from users, specifically relating to the idea itself rather than the kinks of the current design, will be a good benchmark of whether users would play the game on their own when they notice a good opportunity.

Usability Test Procedure

Download a copy of these instructions (.pdf)

The test can be conducted with one or two examiners. The first part of the test aims to evaluate the setup portion of the game. The examiner will need to follow a user as they go through the setup phase, and take notes on what they observe. The second part is two-fold, and is where the test can be separated among two people. To evaluate the game itself, the way a single user interacts with the application is just as crucial as following how the game unfolds among the group. When a user has the phone, notes on their interaction with the application should be taken. The usability goals that relate to the game itself – whether the game is fun, confusing, involves all players, and if the length is adequate – are easier to evaluate from a more removed standpoint, overseeing the group as a whole. Since there are relatively few game screens, a single examiner could potentially do both tasks, just as they could be distributed to two examiners.

Equipment and Handling

As detailed in the User Manual section, the prototype requires an Android smartphone. You may choose to install the application on your own smartphone by following the installation instructions, or we may supply a smartphone with the application on it for testing. From this point, the application is ready to be tested.

Tip: If you are running Android 6.0 or above, you have the option to turn on a feature called Screen Pinning. This allows you limit the use of the phone to the current application only, preventing users from navigating out of the game. This is particularly useful for testing. To turn on Screen Pinning, open the Settings application, navigate to Security, and select “Screen Pinning”. Interacting with Test Subjects

First of all, test subjects should be told that they are testing a rough prototype of the final product, that encountering interaction problems is normal and expected, and that any feedback is appreciated. Test subjects should be given a general idea of the application, and should be able to relate this to their own experience. To do so, the paragraph on our home page serves as a good introduction:

You're sitting with a group of friends, and yet everyone is on their phone. No social interaction whatsoever. Has this ever happened to you? Pass the Phone is a mobile game whose goal may seem contradictory at first: to get people off their phone! The game only needs one person's phone, which gets passed around as people ask questions and answer them as a group.

Since we would like the application to be easy to use, this should be enough exposition on the part of the examiner. The tester should be left alone while using the application, and questions on how to proceed should not be answered (unless the user has been stuck on an issue for a long period of time). This is to avoid interference on the part of the examiner in the process of the game, and to ensure that a user’s feedback in not influenced by outside sources.

Measurements

There are five quantitative measurements, linked to the first five usability goals. Measurements 1 to 3 should be gathered during the setup part, and measurements 4 and 5 relate to the game part.

  1. A new user should make no more than 2 errors throughout the setup.
  2. The setup should take no more than 3 minutes.
  3. The setup screen should be entered in its entirety no more than once for a given group, even if a new round is started.
  4. A new users should ask no more than 3 questions related to the progression of the game
  5. If a player does not receive the phone as often as the other players, or if they do not ask questions as often, then this goal is not met.

Reporting

Free-form notes on the interaction of the users with the game should be taken as the tests are ongoing. Useful things to note include: points of confusion, areas of the application that a user gets stuck on, feedback that they may give, explanations for the reasoning behind an action, perceived strong points and weak points of the interaction, and parts that went smoothly. No rigid structure is imposed on these notes, as they should be taken quickly and spontaneously. They are intended as a qualitative evaluation of the game.

Quantitative measurements relating to the usability goals should be tallied and associated with the number assigned to them. It may be good to organize the five measurements in tabular form. These measurements complete the qualitative notes gathered above to help form a big-picture view of the success of the application with regards to the usability goals that were set.

Heuristic Evaluation

Jakob Nielsen’s list of usability heuristics is used as a base for the heuristic evaluation. Three heuristics were selected at this stage, due to their relation to the usability goals. The evaluators should consider the heuristics and evaluate the degree to which they are met as they navigate through the screens of the setup and the game. Each heuristic is accompanied with a rationale for their inclusion, along with the general principle that should be upheld throughout the prototype for the heuristic to be considered met.

1. Visibility of system status

There are several screens in our application – a user should never feel lost among them. Each screen should explicitly show the user why it exists, conveyings its purpose clearly. Currently, this is addressed used a progress bar during the setup, along with titles on each page throughout. To make the status of the game clear to all players, a text-to-speech component to the game is also being considered, though not implemented in this prototype. This heuristic is central to two of our usability goals: the ease of use of the setup phase, and the flow of the game. Evaluators should consider the clarity of the screen on its own, and in its context. For example, if users lose track of the setup on the third screen, or if the game cannot flow due to confusion, then there is a failure on the part of the application.

2. User control and freedom

The setup phase has several screens, and a single mistake on one of these screens could affect the game afterwards. This heuristics states that a user should be given the tools to undo mistakes or unwanted actions. For this reason, a back button is included throughout the setup. After a game, users can also choose to start a new game by changing only a few settings; these functions should be designed so that a mistake is unlikely, and so that a new game isn't inadvertently started with undesired settings. Evaluators should consider actions that a user could plausibly make by mistake, and see whether it is possible to undo it, or if the application forcibly retains control instead.

3. Match between system and the real world

The application aims to tip-toe the line between digital interaction with a smartphone and real-world interaction with a group of friends. As such, there needs to be a tight coupling between the digital and real world for the actions that the game requires (e.g. physically passing the phone to the next player). The terms chosen should be immediately clear, and be rooted in real-world equivalents. For example, in our low-fidelity prototype, there was some confusion regarding the meaning of the word "group", and why players had to choose a group name at the start of the game. A small explanation was added to elucidate the use of the group name. Evaluators should take note of any other term whose significance is murky, or which is too technical.
