Beta Prototype

Design Evolution

As we expected, the Beta prototype brings many new bug fixes. Additionally, many new features and tweaks have been added to the game. These were found by conducting additional (informal) testing sessions with McGill students. Since the game was much more functional than before, finding improvements to the flow of the game was easier. After the alpha version, we concluded that the setup part mostly satisfied the three usability goals we set for it:

  • 1. it is easy to use;
  • 2. it is not too long; and
  • 3. it does not repeat too often.

Therefore, we decided to focus on the goals for the game itself in the beta version. We had to ensure that:

  • 4. the game flows well from person to person;
  • 5. all players participate in the game;
  • 6. the length of the game is adequate; and
  • 7. the game is a fun solution when people notice that they are all on their smartphone

The following design changes aim to improve the game with these usability goals in mind, along with Nielsen's very helpful list of 10 usability heuristics for UI design.

1. Added Text-to-Speech for portions of the game where it made sense

As we were testing, we noticed that when the phone holder was answering a question, they would naturally keep the four possible choices and their answer to themselves. Since the other players were curious, they always ended up asking the person for this information. We noted that this was an area relating to usability goal 5, and also a special case of the Visibility of system status heuristic.
We were torn between including text-to-speech to voice the contextual information of the game to other players — as was initially included in the sketches developed for the lo-fi prototype — or enticing players to voice the information themselves to encourage interaction, as a traditional board game would require. We opted for the inclusion of text-to-speech because it avoided cluttering the application screen (Aesthetic and minimalist design). Of course, this feature needs to be tested, and possibly assigned an option to toggle it.
For the beta version, we included text-to-speech for the following information, all in areas of the game where a single player may be the only one controlling the flow of the game:

  • "The chosen subject is ..."
  • "To start the game, pass the phone to Jane"
  • "Mike, ask the following question to Alice"
  • "The possible answers for this question are ..."
  • "You answered ..."
  • "This answer is correct/incorrect"

2. Added an option to choose a random subject

This option makes the setup easier to use and shorter (as per goals 2 and 3) when groups of people cannot make their mind as to which subject to choose. The randomizer allows variety and a perception of fairness; it also makes the setup more flexible (Flexibility and efficiency of use).

3. Added a message to indicate who should start with the phone

An oversight to the game was that the person who enters the information for the group in the setup phase may not be the one who is meant to start the game with the phone. In reality, using software tools, there is no surefire way to ensure that the person holding the phone starts first. Therefore, to make the game flow get off on the right foot (usability goal 4), we added a message that tells the first person holding the phone who to pass it to.

The new screen, whose responsibility is to set the game in motion

4. Added a button to ask a teammate for help

To make the game more interactive (usability goal 5), we added a feature to ask a teammate for help with a question. This feature lives alongside the one to ask Google for help. It promotes cooperation, and introduces the potential for memorable moments by sharing the success and failure among multiple people. Each player may use this option once per game. Once the button has been pressed, it becomes inactive, and appears grayed out to convey that status. Clicking on it again will make a message appear, explaining that the option is only available once per game, per player. Also, this option only appears in team mode, since it wouldn't make sense to ask opposing players for help.

Top: new button to ask a friend for help; bottom: grayed out button once the option has been used

5. Added a progress bar for the game

The setup part of the game already had a progress bar, to ensure the Visibility of system status. We decided to add one for the game as well, to allow players to know at a glance how far along they are. Currently, each player gets two turns; after the tests that have been conducted, we believe this gives the game an adequate duration (usability goal 6)

The newly-added progress bar in the game screen

6. Added a loading screen

Due to NativeScript, the framework we used to ensure that the game could run on both iOS and Android, the game needs to load for a non-negligible period of time before it can start. We decided to add a loading screen during this time, to ensure that users understand why the game is unresponsive. This is in line with two heuristics: Match between the system and the real world, because the screen uses the widely known convention behind the meaning of the word "loading", and Visibility of system status, because it allows users to know why they are made to wait.

Pass the Phone's new loading screen

7. Further error checking and prevention in the team-building phase

The team-building phase has been the focus of many of our efforts at Error prevention, one of Nielsen's 10 heuristics. Two new cases where mistakes could be made are now dealth with, and the application now displays appropriate error messages. Users can no longer choose duplicate names, and a game can no longer be started by a single person. These improvements also make the setup easier to use (goal 1).

Left: error message if two players try to use the name Mike; right: error message if Mike tries to create a group with only himself in it

Callback to the goal of the project

This is the final prototype in the scope of the class, and so it is an appropriate time to reflect on whether we accomplished our goal. From our low-fidelity prototype:

The task is to design a mobile game that facilitates face-to-face social interaction in a group of people that know each other, all the while minimizing isolation when interacting with the app.

In this context, the design challenge can be refined into two main components. The first one is to minimize the complexity and setup time of the game. This is to allow people to play as soon as possible, so they can spend less time isolated on their phone. The second one is more complex to achieve, because it’s about using the phone to get people off of their phone. While building the game component of the application, it must be simple enough so that the users can easily go through the game while having social interaction with others. Also, it should encourage social interaction by enticing players to interact with each other to achieve their goal in the game.

We believe that the communication goal at the root of Pass the Phone is well met by our current implementation. There could be more testing and refinement, but the current game is a good solution to stimulate interaction among a group of friends. The iterative process definitely allowed us to move forward at each step, gradually solving usability problems and moving closer to the ideal usability goals that we set for the project.
